Activity四种启动模式

1. standard: 标准模式，也是默认模式

每次启动都会重新创建一个新的实例，不管这个实例是否已经存在。

2. singleTop: 栈顶复用模式。

如果新Activity已经位于任务栈的栈顶，那么Activity不会被重新创建，同时它的onNewIntent方法会被调用。这个Activity的onCreate，onStart不会被系统调用。如果新的Activity的实例已经存在但不是位于栈顶，那么新Activity仍然会重新重建。

3. singleTask: 栈内复用模式。

这是一种单实例模式，在这种模式下，只要Activity在一个栈中存在，那么多次启动此Activity都不会重新创建实例，和singleTop一样，系统也会回调其onNewIntent，且将其调到栈顶，而且会带有clearTop效果，即将栈内所有在它上面的Activity全部出栈。

4. singleInstance: 单实例模式。

一种加强的singleTask模式，具有此种模式的Activity只能单独位于一个任务栈中。

Java内部类和静态内部类

类

Java类实例化时，执行顺序：

  1. 静态变量 / 静态代码块
  2. 非静态变量 / 代码块
  3. 构造方法

有父类，子类实例化的时候，顺序：

  1. 父类的静态变量 / 静态代码块
  2. 子类的静态变量 / 静态代码块
  3. 父类的非静态变量 / 代码块
  4. 父类的构造方法
  5. 子类的非静态变量 / 代码块
  6. 子类的构造方法

内部类

什么是内部类？

可以将一个类的定义放在另一个类的定义内部，这就是内部类。

【特点】：

当生成一个内部类的对象时，此对象与制造它的外围对象之间就有了一种联系，所以内部类能访问其外围对象的所有成员，而不需要任何特殊条件，还拥有其外围类的所有元素访问权。

.this & .new

内部类生成对外部类对象的引用，可以使用外部类类名后紧跟.this。

创建内部类对象，必须使用外部类的对象来创建该内部类对象。

在拥有外部类对象之前是不可能创建内部类对象的。

内部类与向上转型

当将内部类向上转型为其基类，尤其是转型为一个接口的时候，内部类就有了用武之地。

因为，此内部类 —> 某个接口的实现 —> 能够完全不可见，并且不可用。

所得到的只是指向基类或接口的引用，所以能够很方便地隐藏实现细节。

嵌套类（静态内部类）

如果不需要内部类对象与其外围类对象之间有联系，那么可以将内部类声明为static，通常称为嵌套类（静态内部类）

普通的内部类对象隐式地保存了一个引用，指向创建它的外围类对象。

但是，当加上static后：

  1. 要创建嵌套类的对象，并不需要其外围类的对象。
  2. 不能从嵌套类的对象中访问非静态的外围类对象。

普通的内部类不能有static数据和static字段，也不能包含嵌套类。但是嵌套类可以包含所有这些东西。

为什么要使用内部类？

使用内部类最吸引人的原因：
每一个内部类都能独立的继承自一个（接口的）实现，所以无论外围类是否已经继承了某个（接口的）实现，对于内部类都没有影响。

内部类有效的实现了”多重继承”,也就是说，内部类允许继承多个非接口类型（类或抽象类）。

内部类标识符

每个类都会产生一个.class文件，其中包含了如何创建该类型的对象的全部信息。内部类也不例外，同样必须生成一个.class文件以包含它们的Class对象信息。命名的规则是：外围类的名字，加上“$”，再加上内部类的名字。

如果是匿名内部类，编译器会简单的产生一个数字作为其标识符。

Android Jetpack - LiveData

Android Jetpack: 一个Android软件组件的集合，包含了开发库，工具，以及最佳实践指南。

Lifecycle库：有效避免内存泄漏和解决Android生命周期的常见难题。

LiveData：是一种具有生命周期感知能力的可观察数据持有类。

如何添加组件

LiveData 与实现了LifecycleOwner接口的对象配合使用，我们现在常用的 AppCompatActivity, Fragment都是实现了这个接口。

简而言之

  1. LiveData通过应用观察者模式，可以使屏幕上显示的内容与数据随时保持同步。

  2. 其具有生命周期感知能力，指当Lifecycle Object状态变为DESTROYED的时候，会自动移除observer。

    并且，只在 STARTED 或者 RESUMED 状态时，LiveData才会通知界面更新。

    即在页面活跃的时候才会有通知，当页面在不可见的时候不会产生不必要的更新通知，而一旦页面回到可见状态会立即受到最新的数据。

具体的讲LiveData的好处：(从官方文档抄的，翻译的…大概就自己能懂) 文档地址

  1. 确保UI与数据状态的匹配

    生命周期变更时，LiveData会通知对象。通过观察者模式实现了响应式的UI数据更新。

  2. 有效避免内存泄漏

    观察者保定Lifecycle对象，在Lifecycle对象生命周期结束后（DESTROYED）自行清理。

  3. 不会因停止Activity而引起崩溃

    LiveData能感知生命周期，当UI处于不活跃状态时候，不会发出更新通知。

  4. 不用手动处理生命周期

    UI 控件只需要观察相关数据，不需要停止或恢复观察。这些LiveData会自动处理，因为它能感知生命周期的变化。

  5. 总是保持最新的数据

    如果生命周期变为非活动状态，则会在再次变为活动状态时接收最新数据。例如，后台Activity在返回前台后会立即收到最新数据。

  6. 适应配置更改

    当一个Activity或者Fragment由于配置改变被重建的时候，例如：设备旋转，它会立即收到最新的可用数据。

  7. 共享资源

    你可以使用一个继承自LiveData的对象，使用单例模式，此对象包装了一个系统服务，这样就可以在App内共享服务。只要LiveData与这个System Service建立一次连接，这样其有需求的观察者只要观察这个LiveData即可。
