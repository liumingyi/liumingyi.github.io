5-单件模式

单件模式：确保一个类只有一个实例，并提供全局访问点。

个人感受：这是一个常见但可能不常写的模式。很简单很有用也很重要。

不废话，也不画图了，直接代码。不谈论线程不安全的写法！！！

以下方式都是线程安全的。

方式一：急切创建

instance 在类装载时就初始化，尽管大部分时候都是在调用 getInstance()时候被装载，但并不完全。

在没有硬性要求延迟加载时使用这种方式就可以了，没有加锁，执行效率高。

缺点：容易产生垃圾，浪费内存。

方式二：延迟创建

1.静态内部类。由方式一进化而来。

使用一个静态内部类，只有在调用getInstance()时候这个内部类才被装载，instance才被创建，由此达到延迟创建效果。

2.加锁

首先. 方法加锁，简单粗暴。直接对整个instance方法加锁。

此方法可能造成程序执行效率下降百倍。

因为，这把锁只有在第一次创建的时候有意义，之后都变成了累赘！

其次. 双重检查加锁。

volatile关键字确保：当uniqueInstance变量被初始化成Singleton实例时，多个线程正确地处理uniqueInstance变量。

方式三：涉及反序列化创建对象时，枚举

前面两种方式，在反序列化后都会失效。

而这一种能防止反序列化后重新创建新的对象。

Android屏幕适配

尺寸单位：

参考Android官方文档

参考2

px (pixel): 像素。 （设计师给的设计图通常都是以此为单位。）

dpi (dots per inch): 像素密度，每英寸的像素点数。

dp (density-independent pixel): 同时也叫做dip，密度无关像素，与终端上的实际物理像素点无关。密度无关像素等于 160 dpi 屏幕上的一个物理像素。

注意哦：dp即dip，dip不是dpi，切勿混淆，为了不混淆还是忘掉dip吧，它就是dp！

另外物理尺寸：1英寸 = 2.54 cm

平时所说的手机屏幕尺寸是指的对角线长度。而 “像素密度 = 对角线像素 / 对角线的物理尺寸”。

非要弄个公式的话: $$屏幕像素密度(dpi) = \frac{\sqrt{长(px)^2 + 宽(px)^2}}{尺寸(inch)}$$

比如：Samsung s7 edge手机，屏幕尺寸5.5英寸，分辨率2560 x 1440像素。

$$s7的像素密度 = \frac{\sqrt{2560^2 + 1440^2}}{5.5} \approx 534 $$

为简便起见，Android 将所有屏幕密度分组为六种通用密度： 低、中、高、超高、超超高和超超超高

密度 屏幕像素密度
ldpi（低） ~120dpi
mdpi（中） ~160dpi
hdpi（高） ~240dpi
xhdpi（超高） ~320dpi
xxhdpi（超超高） ~480dpi
xxxhdpi（超超超高） ~640dpi

如何适配

  1. 不必说，当然是使用 dp ，而不是像素为单位，忘掉AbsoluteLayout（很久很久以前就已被官方弃用）
  2. 多用RelativeLayout，LinearLayout，ConstraintLayout，善用wrap_content,match_parent,weight来控制视图组件的宽度和高度。
  3. 使用限定符，根据屏幕的配置来加载相应的UI布局。
  4. 配置不同分辨率的图片资源，使用自动拉伸的.9图，使图片资源自适应屏幕尺寸。

4-工厂模式

工厂模式有两种：工厂方法模式 & 抽象工厂模式。

有时候会说到简单工厂，其实简单工厂不是一个模式，而更像是一种编程习惯。将产品类的实例化纳入一个工厂类，然后该工厂类提供一个静态的创建方法。

工厂方法模式：定义了一个创建对象的接口，但由子类决定要实例化的类是哪一个。工厂方法让类把实例化推迟到子类

抽象工厂模式：提供一个接口，用于创建相关或依赖对象的家族，而不需要明确指定具体类型

工厂方法和抽象工厂的对比

首先，所有的工厂都是用来封装对象的创建。通过减少应用程序和具体类之间的依赖促进松耦合

工厂方法：使用继承把对象的创建委托给子类，子类实现工厂方法来创建对象。

抽象工厂：使用对象组合，对象的创建被实现在工厂接口所暴露出来的方法中。创建相关的对象家族，而不需要依赖他们的具体类。

2.观察者模式

观察者模式（Observer Pattern）

“在对象之间定义一对多的依赖，这样一来，当一个对象改变状态，依赖它的对象都会收到通知，并自动更新”

例子：天气显示板，天气变化时通知所有显示板更新数据。

Android中，观察者模式实在是太多了。

比如：点击事件，广播，adapter
